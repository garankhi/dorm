# SDMS MVP EF + Supabase Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build a small .NET Web API + EF Core dormitory MVP using Supabase PostgreSQL, around the flow Student applies, Admin approves, Contract and Invoice are created.

**Architecture:** Frontend calls only the .NET 8 Web API. The API uses JWT auth, Entity Framework Core with Npgsql, and a Supabase PostgreSQL connection string. `/login` redirects users into separate `/admin/*` and `/student/*` portals based on the role returned by the API.

**Tech Stack:** .NET 8 Web API, Entity Framework Core, Npgsql PostgreSQL provider, Supabase PostgreSQL, JWT Bearer Auth, frontend selected by the team.

### Task 3: Build Frontend Portal Routing

**Owner:** Nhóm 1

- [ ] **Step 1: Implement login/register/logout UI**

Frontend calls backend API only. Do not use Supabase JS client for database or auth.

- [ ] **Step 2: Load current user**

Use JWT plus `GET /api/auth/me` to get current user and role.

- [ ] **Step 3: Create portal routes**

Create these frontend route groups:

```text

/admin/*
/student/*
```

`/admin/*` uses the Admin layout/menu. `/student/*` uses the Student layout/menu.

- [ ] **Step 4: Route by role**

After login, if role is `admin`, redirect to `/admin`. If role is `student`, redirect to `/student`.

### Task 4: Build Registration and Room Flow

**Owner:** Nhóm 2

- [ ] **Step 1: Build room APIs**

Create:

```text
GET /api/student/rooms
GET /api/admin/rooms
POST /api/admin/rooms
PUT /api/admin/rooms/{id}
```

Student room list returns rooms where `status = 'available'` and `current_occupancy < capacity`.

- [ ] **Step 2: Build application APIs**

Create:

```text
POST /api/student/dorm-applications
GET /api/student/dorm-applications/my
GET /api/admin/dorm-applications?status=pending
POST /api/admin/dorm-applications/{id}/reject
```

Student insert uses the current user id from JWT, not from client-submitted `studentId`.

- [ ] **Step 3: Build frontend screens**

Student screens: room list, create application, my applications.

Admin screens: room management, pending application list, reject application.

### Task 5: Build Approval Transaction, Contract and Billing

**Owner:** Nhóm 3

- [ ] **Step 1: Implement approve service transaction**

Create backend service method equivalent to:

```text
ApproveApplication(applicationId, adminUserId, startDate, contractMonths)
```

Inside one EF transaction:

```text
1. Load pending application.
2. Load room.
3. Reject if room is full, maintenance, or inactive.
4. Set application status = approved.
5. Create contract.
6. Increment room current occupancy.
7. Set room status = full if occupancy reaches capacity.
8. Create first invoice.
9. Save changes and commit.
```

- [ ] **Step 2: Expose approve API**

Create:

```text
POST /api/admin/dorm-applications/{id}/approve
```

Only Admin can call it.

- [ ] **Step 3: Build contract APIs**

Create:

```text
GET /api/student/contracts/my
GET /api/admin/contracts
```

- [ ] **Step 4: Build invoice APIs**

Create:

```text
GET /api/student/invoices/my
GET /api/admin/invoices
POST /api/admin/invoices/{id}/mark-paid
```

`mark-paid` creates a confirmed payment row and sets invoice status to `paid`.

- [ ] **Step 5: Build contract/invoice frontend screens**

Student sees own contract and invoices. Admin sees invoices and can mark paid.
